# eaglem_team
# 🚚 Прогнозирование загрузки грузовых машин (Хакатон РСХБ)

## 🎯 Бизнес-задача

Цель  
Спрогнозировать целевую метрику target_2h (характеризующую загрузку грузового автомобиля) на 10 шагов вперёд (каждый шаг = 30 минут) для команды team.

Бизнес-ценность  
Точный прогноз загрузки транспорта позволяет:
- 🗺️ Оптимизировать маршруты и графики доставки.
- 📉 Снизить операционные расходы за счёт эффективного распределения ресурсов.
- 🤝 Повысить качество обслуживания клиентов благодаря предсказуемости поставок.

---

## 🏗️ Архитектура системы

### Компоненты
| Компонент | Технология | Порт | Функции |
|:---|:---|:---:|:---|
| API-сервер | C++ (Crow + httplib) | 8080 | Принимает JSON от клиента, валидирует данные, вызывает ML-сервис, выполняет жадный подбор транспорта, формирует итоговый ответ. |
| ML-сервис | Python (Flask) | 5000 | Загружает предобученную модель (`model.joblib`), принимает фичи от C++-сервера, генерирует прогноз объёма отгрузки на 2 часа, возвращает JSON. |

### Взаимодействие компонентов
1. Клиент отправляет `POST`-запрос на C++-сервер (порт `8080`) с данными о статусах обработки товаров.
2. C++-сервер валидирует входные данные.
3. C++-сервер формирует `HTTP POST`-запрос к Python-сервису (порт `5000`) для получения прогноза объёма.
4. Python-сервис загружает модель, выполняет предсказание и возвращает объём в формате JSON.
5. C++-сервис на основе прогноза выполняет оптимизацию подбора транспорта из доступного парка.
6. C++-сервер возвращает клиенту итоговый JSON со списком машин и сопутствующими метриками.

### Алгоритм подбора транспорта
1. Фильтрация: выбираются только машины со статусом available: true.
2. Сортировка: парк сортируется по вместимости (по умолчанию) или по стоимостной эффективности.
3. Жадный выбор: машины последовательно добавляются в набор, пока суммарная вместимость не покроет прогнозный объём отгрузки.

### Бизнес-допущения
- Груз делим между машинами без ограничений.
- Габариты груза не учитываются (гарантированно помещается в машину).
- География ограничена одним складом, маршрутизация не производится — выполняется только расчёт количества транспорта.

## 🧠 ML Pipeline: Обучение и прогнозирование

### 1. Загрузка и первичный анализ (EDA)
- Данные загружаются с Google Drive с помощью gdown.
- Проверка на пропуски: данные полные, импутация не требуется.
- Анализ распределений признаков (`status_1`–`status_8`) и целевой переменной target_2h.
- Визуализация корреляций подтверждает высокую значимость статусных признаков для прогноза.

### 2. Подготовка данных (Feature Engineering)
- Сдвиг таргета: для каждого route_id создаются колонки target_step_1`…`target_step_10 путём сдвига исходной target_2h на 1–10 шагов вперёд.
- Отбор строк: из обучающей выборки удаляются строки, где отсутствуют полные будущие таргеты (`dropna`).
- Временное окно: для обучения используются только последние 14 дней данных (`2025-05-16` – `2025-05-30`), чтобы модель адаптировалась к актуальному режиму работы.
- Train/Valid split: временная стратификация — 80% данных (ранние по времени) идут на обучение (`fit`), 20% (поздние) — на валидацию.

### 3. Признаки (Features)
| Тип | Признаки | Метод обработки |
|:---|:---|:---|
| 🔵 Категориальные | office_from_id, route_id | OneHotEncoder |
| 🟢 Числовые | status_1 … status_8 | StandardScaler |

### 4. Моделирование

#### 📦 Бейзлайн: Ridge-регрессия (линейная)
- Почему Ridge? Устойчива к мультиколлинеарности, корректно работает с OneHot-кодированием, обеспечивает интерпретируемость коэффициентов.
- Гиперпараметры: alpha = 4.0 (подобрано вручную).
- Мультитаргет: модель обучается сразу на 10 шагов (multi-output regression).

- Архитектура на основе Transformer, адаптированная для временных рядов.
- Суть: каждая временная точка кодируется в вектор, attention работает по временной оси.
- В коде приведён пример обучения на выборке 60k строк (для демонстрации). Полное обучение на всём датасете требует больших вычислительных ресурсов.

### 5. Метрика качества
Используется кастомная метрика WAPE + Relative Bias
